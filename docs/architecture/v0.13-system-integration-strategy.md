# v0.13 System Integration Strategy

## Executive decision

The selected v0.13 system strategy is:

```text
PostgreSQL control plane with evidence-gated specialized derived planes.
```

The goal is not to add engines for their own sake. The goal is to isolate workload classes while keeping one authoritative command, permission, audit, outbox, and lineage boundary.

## Selected system map

| Plane | Role | Owns truth for | Consumes from | Writes to |
|---|---|---|---|---|
| PostgreSQL control plane | Commands, business objects, permissions, workflow, audit, outbox, projections | operational ERP truth and metadata | API/domain handlers | PostgreSQL only |
| TigerBeetle numeric ledger plane | Conserved numeric movement after P1 adoption | ledger-derived balances after cutover | NumericLedgerPort movement plan | TigerBeetle, then PostgreSQL projection repair |
| pgvector semantic retrieval plane | Similarity search over approved chunks | nearest-neighbor retrieval result only | permissioned chunks | vector indexes only |
| DuckDB analytics plane | Analytical reports, exports, reconciliation | derived analytical result artifacts only | governed snapshots/Parquet/projections | result artifacts and query-job status |
| Formula worker plane | Spreadsheet formula computation | formula result projections | formula graph/deltas | formula result projection path |
| Outbox/SSE delivery plane | Live delivery | delivery envelope state | PostgreSQL outbox | client delivery state |

## Strategy alternatives

### A. PostgreSQL-only monolith

**Score:** 6.6/10.

PostgreSQL runs OLTP, analytics, vectors, ledger rows, formulas, audit, outbox, and exports.

**Strengths:** simple backups, fewest systems, easiest local development.

**Weaknesses:** analytical scans and vector indexes compete with edit latency; numeric ledger invariants remain custom; scaling pressure concentrates on one database.

**Decision:** keep MVP-compatible, reject as post-MVP target.

### B. PostgreSQL extensions only

**Score:** 7.3/10.

Use PostgreSQL plus pgvector and pg_duckdb-style extension paths.

**Strengths:** keeps SQL surface close to PostgreSQL and may reduce data movement.

**Weaknesses:** weak isolation between OLTP and analytics if not carefully separated; extension maturity and operational limits become production risks.

**Decision:** evaluate selectively after P1 evidence. Not the default product architecture.

### C. Hub-and-spoke specialized planes

**Score:** 9.3/10. **Selected.**

PostgreSQL remains control/projection/audit hub. TigerBeetle, pgvector, and DuckDB consume explicit contracts and publish derived results or projections back through PostgreSQL lineage.

**Strengths:** workload isolation, clear source-of-truth boundaries, adapter-based migration, strong compliance story.

**Weaknesses:** more contracts, more runbooks, more observability.

### D. Event-stream-first architecture

**Score:** 7.5/10 later, 4.8/10 for MVP.

A stream/event lake becomes the primary integration backbone.

**Strengths:** good for large integrations and replay.

**Weaknesses:** too much platform before the command/outbox foundations are proven. Current durable outbox already provides the safer first step.

**Decision:** defer.

### E. Warehouse/lakehouse-first architecture

**Score:** 8.0/10 later, 4.5/10 for MVP.

Operational data is continuously exported to analytical storage first.

**Strengths:** strong BI and long retention.

**Weaknesses:** too heavy for post-MVP first pass and risks duplicating ERP semantics outside command lineage.

**Decision:** DuckDB/Parquet snapshots keep this path open without committing now.

### F. Microservice database per capability

**Score:** 5.5/10.

Split inventory, finance, workflow, analytics, AI, and formula capability into separate services early.

**Strengths:** team autonomy at scale.

**Weaknesses:** distributed transaction complexity, uncertain ownership, hard Phase 0 debugging, and early cognitive load.

**Decision:** reject until boundaries and invariants have production evidence.

### G. AI-agent orchestration first

**Score:** 4.2/10.

Agents dynamically choose between PostgreSQL, TigerBeetle, pgvector, DuckDB, and formula tools.

**Strengths:** attractive UX.

**Weaknesses:** unsafe until deterministic permission, command, and data-plane contracts are mature.

**Decision:** AI may assist retrieval and explanation; it cannot be the integration authority.

### H. Client/edge analytics first

**Score:** 6.0/10.

Use DuckDB/WASM or local snapshots for rich client analytics.

**Strengths:** good for offline support, power-user exports, and local profiling.

**Weaknesses:** data leakage, versioning, resource, and supportability risks.

**Decision:** later feature path only.

## Selected pattern in one rule

```text
Control stays centralized; computation may specialize; all specialization is derived, permissioned, versioned, and reversible.
```

## Why this is synergistic

- PostgreSQL gives every plane consistent identity, tenancy, permissions, audit, and lineage.
- TigerBeetle removes custom balance-race logic from conserved numeric movement.
- pgvector narrows semantic context without deciding truth or authorization.
- DuckDB computes deterministic aggregates over governed snapshots without harming OLTP edits.
- Formula workers keep spreadsheet computation resident and incremental.
- Outbox polling makes derived work replayable and recoverable.

## pgvector strategy refinement

The pgvector plane uses the selected model from `docs/data/pgvector-integration-strategy-options.md`:

```text
permissioned derived chunk registry
+ hybrid lexical/vector retrieval
+ evidence-gated ANN indexes
+ optional dedicated semantic PostgreSQL/pgvector database after scale evidence
```

This means pgvector is not a broad AI query router and not an authorization or mutation plane. It narrows context; deterministic PostgreSQL, TigerBeetle-derived projections, and DuckDB analytics provide facts. User-visible retrieval requires permission, classification, and source-version revalidation.


## Outbox fan-out strategy refinement

PostgreSQL transactional outbox remains authoritative. MVP uses polling/SSE and internal workers. Post-MVP fan-out may add an internal dispatcher, CDC shadow bridge, broker streams, or managed event buses only after P1-OUTBOX-001 evidence.

Canonical docs: `docs/data/event-envelope-contract.md`, `docs/data/outbox-integration-strategy-options.md`, `docs/adr/ADR-0024-post-mvp-outbox-fanout-and-cdc-strategy.md`, and `docs/gates/P1-OUTBOX-001-post-mvp-outbox-fanout-cdc-spike.md`.

Synergy rule: all specialized planes receive invalidation, scheduling, and replay through governed outbox envelopes or derived manifests. No specialized plane infers business events from raw table changes.

## v0.14 external integration extension

External systems are added as a post-MVP connector plane. They do not become a source of operational mutation authority.

```text
Inbound external data -> integration intake/staging -> command proposal/submission -> command handler
Outbound external data -> committed outbox envelope -> integration dispatcher -> external system
```

This preserves PostgreSQL command/outbox authority while allowing later integrations with SaaS APIs, webhooks, SCIM, file/batch exchange, DuckDB snapshots, pgvector content feeds, and TigerBeetle reconciliation workflows.
